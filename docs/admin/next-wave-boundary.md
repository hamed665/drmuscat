# Next Wave Boundary

This document defines the boundary after the first controlled provider public-state step.

A passing first packet proves one provider path only. It does not approve a wider provider wave.

## Allowed mode

The only approved mode is one provider at a time.

Each provider must have its own complete packet:

- readiness bundle review
- candidate selection worksheet
- rehearsal pattern applied to that provider
- operator checklist
- go / no-go decision record
- observation report after the final admin step

No provider may reuse another provider packet or decision.

## Not allowed

The following shortcuts are not allowed:

- multi-provider public-state change
- shared decision across multiple providers
- skipped worksheet
- skipped rehearsal
- skipped operator checklist
- skipped observation report
- using one successful provider as proof for the next provider
- manual database edits for public state
- manual sitemap edits
- bypassing the public eligibility wrapper

## Per-provider decision rule

Every provider must pass its own evidence chain before the final gated admin control is used.

If a provider has unclear location data, unapproved taxonomy, quality blockers, unresolved contact visibility, unsupported public copy, audit uncertainty, or indexability risk, that provider is No-Go until fixed.

## Operator boundary

Operators must not convert the controlled path into a campaign or batch workflow.

The active centers read-only view and audit log are review surfaces only. They are not rollout tools and they must not grow edit controls under this boundary.

## What requires a new contract

These require a separate contract, implementation, audit behavior, and validator before use:

- multi-provider public-state workflow
- active provider editing
- claim workflow
- billing or subscription workflow
- sponsored ranking
- review or rating workflow
- open-now claims
- booking claims
- insurance claims
- automated sitemap promotion outside existing guards

## Pass criteria

The boundary is respected only when:

- the process remains one provider at a time
- every provider has its own packet
- every provider has its own Go decision
- every public-state change uses only the final gated admin control
- every public-state change is followed by an observation report
- public visibility continues to depend on the eligibility wrapper
- no manual database or sitemap edits are used
