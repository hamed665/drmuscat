# DrMuscat Import Readiness + Controlled Publishing Architecture V1

Status: documentation-only planning contract  
Build mode: PHASED_BUILD_ONLY  
Scope: import readiness, controlled publishing, SEO validation, geo validation, domain separation, internal linking, sitemap eligibility, schema validation, admin readiness, and performance guardrails.

This document is the canonical implementation plan for keeping imported healthcare entities private until they are validated, approved, link-safe, schema-safe, and sitemap-safe. It does not approve new public surfaces by itself.

## Core rule

No entity becomes public only because it exists in the database.

Every imported or created entity must move through a controlled lifecycle:

```text
Imported / Created
↓
Private + Noindex + Sitemap Excluded
↓
Domain checked
↓
Canonical geo checked
↓
SEO checked
↓
Content checked
↓
Duplicate checked
↓
Schema generated and validated
↓
Internal links generated into cache
↓
Manual approved
↓
Public
↓
Sitemap eligible
```

## Non-negotiable invariants

1. Import is never publish.
2. Imported CSV rows must default to private visibility.
3. Sitemap eligibility is separate from publish eligibility.
4. Human healthcare and pet healthcare must not cross-link by default.
5. Internal links must be rule-based and precomputed, not generated by heavy runtime queries on public page render.
6. Canonical geo records must be used instead of free-text city and area values.
7. Public pages must read only page-ready data, precomputed schema, and cached internal links.
8. Admin/import surfaces must remain separate from public surfaces.

## PR 1: Import Publish Lock

Goal: make every imported entity private and non-indexable by default.

Required defaults for every CSV/import-created entity:

```text
visibility = private
index_policy = noindex
sitemap_policy = excluded
manual_approved = false
```

Implementation requirements:

- Import code must not accept public visibility from CSV input.
- Import code must not infer sitemap inclusion.
- Import code must not set manual approval.
- Import dry-runs must report the default private/noindex/excluded state.
- Tests must prove imported rows cannot become public directly.

Expected output:

- Private import queue baseline.
- No imported entity enters the public catalog only from import execution.
- Admin/dry-run reporting shows import state separately from publish state.

## PR 2: Entity Lifecycle + Publish Eligibility

Goal: centralize lifecycle and publish readiness.

Recommended lifecycle states:

```text
draft
imported
validation_failed
validated
approved
published
unpublished
archived
```

Required central functions:

```ts
canPublishEntity(entity)
getPublishBlockers(entity)
getReadinessStatus(entity)
```

Rules:

- `public_ready` should be computed from validation state, not manually trusted as source-of-truth.
- If a cached readiness snapshot is needed for performance, it must be treated as a snapshot, not authority.
- Publish mutations must call the central eligibility function.
- Public routes must not bypass eligibility.

Expected blockers include:

```text
missing_domain
invalid_entity_type
missing_canonical_geo
low_geo_confidence
missing_slug
invalid_title
missing_meta_description
canonical_invalid
duplicate_risk
content_incomplete
schema_missing
schema_invalid
relations_missing
manual_approval_missing
```

## PR 3: Domain + Entity Type Contract

Goal: prevent human healthcare, pet healthcare, beauty, wellness, and fitness from becoming one SEO soup.

Required domains:

```text
human_healthcare
pet_healthcare
medical_beauty
non_medical_beauty
wellness
fitness
```

Domain separation rules:

- `human_healthcare` must not link to `pet_healthcare` by default.
- `pet_healthcare` must not link to `human_healthcare` by default.
- `medical_beauty` and `non_medical_beauty` must be distinct.
- `fitness` may only link to healthcare entities through explicit low-priority rules.
- Domain must be enforced in link rules, URL planning, breadcrumb generation, schema generation, and article cluster assignment.

Example human healthcare allowed targets:

```text
doctor
pharmacy
lab
imaging_center
dental_clinic
medical_beauty_clinic
physiotherapy
nearby_hospital
```

Example human healthcare blocked targets:

```text
pet_shop
pet_clinic
pet_pharmacy
pet_grooming
pet_boarding
```

Example pet healthcare allowed targets:

```text
vet_doctor
pet_clinic
pet_pharmacy
pet_shop
pet_grooming
pet_boarding
nearby_pet_clinic
```

Example pet healthcare blocked targets:

```text
human_doctor
hospital
human_pharmacy
human_lab
human_imaging_center
```


```text

```

Recommended geo resolution states:


Rules:

- Free-text city/area may be stored as raw import evidence, but must not drive public URLs, sitemap output, or internal linking.

## PR 5: SEO / Content / Schema Validation Blockers

Goal: block publishing before core public-page quality is ready.

Required validation gates before publish:

```text
slug valid
title valid
meta description valid
canonical valid
domain valid
entity type valid
canonical geo valid
minimum content complete
duplicate check passed
schema generated
schema valid
internal links generated
manual approved
```

If any gate fails:

```text
publish blocked
reason shown in admin
```

This PR should not create new public routes. It only defines validation contracts and blockers.

## PR 6: Link Rule Matrix

Goal: make internal linking deterministic and safe.

Required table or config model:

```text
entity_link_rules
```

Required columns:

```text
source_type
target_type
source_domain
target_domain
allowed
priority
max_links
max_distance_km
same_city_required
same_area_boost
same_specialty_required
min_quality_score
```

Required example rules:

```text
hospital -> doctor = allowed
hospital -> pharmacy = allowed
hospital -> lab = allowed
hospital -> imaging_center = allowed
hospital -> pet_shop = blocked
pet_clinic -> pet_shop = allowed
pet_clinic -> pet_pharmacy = allowed
pet_clinic -> human_pharmacy = blocked
medical_beauty_clinic -> dermatologist = allowed
dental_clinic -> dentist = allowed
```

Rules:

- Blocked rules must win over broad allow rules.
- Cross-domain rules require explicit approval.
- Same-area and same-city boosting must never override a blocked domain pair.

## PR 8: Sitemap Eligibility

Goal: sitemap should include only validated, indexable, approved public records.

Required central function:

```ts
isSitemapEligible(entity)
```

Required sitemap eligibility gates:

```text
visibility = public
index_policy = index
sitemap_policy = included
seo_validated = true
geo_validated = true
content_validated = true
schema_validated = true
relations_validated = true
duplicate_check_passed = true
manual_approved = true
```

Rules:

- Sitemap generation must not read from raw entity tables directly.
- Sitemap generation should read from a validated projection or equivalent query boundary such as `public_indexable_entities`.
- An entity can be public but still excluded from sitemap if sitemap eligibility fails.

Recommended sitemap files:

```text
/sitemaps/doctors.xml
/sitemaps/pharmacies.xml
/sitemaps/hospitals.xml
/sitemaps/clinics.xml
/sitemaps/dental.xml
/sitemaps/beauty.xml
/sitemaps/pet.xml
/sitemaps/locations.xml
/sitemaps/articles.xml
```

```text

```

```text

```

Pet clinic pages:


Rules:

- Pet pages must not receive human medical schema.
- Human healthcare pages must not receive pet schema.
- Schema validation failure blocks publish and sitemap eligibility.
- Generated schema should be precomputed or stored in a lightweight projection for public rendering.

## PR 11: Manual Publish Flow

Goal: publishing must happen through one explicit controlled flow.

Allowed publish sequence:

```text
Validate
→ Generate internal links
→ Generate schema
→ Check sitemap eligibility
→ Manual approve
→ Publish
```

Rules:

- Publish action must call `canPublishEntity()`.
- Sitemap inclusion must call `isSitemapEligible()`.
- Manual approval must be recorded with actor and timestamp.
- Failed publish attempts should return blockers, not partially publish.
- Unpublish must reverse public visibility and sitemap inclusion without deleting import evidence.

## PR 12: Performance Guard

Goal: public profile pages must stay fast and avoid heavy runtime work.

Rules:

- Public pages should read only page-ready data.
- Internal links must be read from cache.
- Schema should be precomputed or lightweight.
- Sitemap runtime must not run heavy validation or geo queries.
- No heavy distance calculation should happen during public page render.
- Admin/import workflows may perform heavy validation outside the public render path.

Performance target:

```text
Public pages should remain under 2 seconds for TTFB/render budget under normal production conditions.
```

Recommended public read path:

```text
entity profile projection
precomputed schema
entity_internal_links_cache
public_indexable_entities or equivalent validated projection
```

## Execution order

The implementation order must remain:

```text
1. Import Publish Lock
2. Entity Lifecycle + Publish Eligibility
3. Domain + Entity Type Contract
4. Canonical Geo
5. SEO / Content / Schema Blockers
6. Link Rule Matrix
7. Internal Link Generator + Versioned Cache
8. Sitemap Eligibility
9. Schema Generator + Validation
10. Admin SEO Readiness Panel
11. Manual Publish Flow
12. Performance Guard
```

Do not start with internal linking, schema expansion, or sitemap expansion before import publish lock and entity lifecycle are in place.

## Definition of done for the architecture wave

This architecture wave is complete only when:

- imports default to private/noindex/excluded,
- publish eligibility is centralized,
- sitemap eligibility is centralized and separate,
- canonical geo is required for public records,
- human/pet/beauty/wellness/fitness domain separation is enforced,
- internal links are generated through rules and cached,
- schema is generated and validated per domain/type,
- admin can explain readiness blockers,
- manual publish is the only publish path,
- public page render does not perform heavy geo/link generation work.
